Linux Logs

Linux System Logs

Logs play a critical role in Linux systems, serving as a central source of information about system operations, application performance, and security incidents. These logs are managed by tools like syslog daemons (e.g., rsyslog, syslog-ng) and systemd's journald, which provide robust logging functionalities tailored to the needs of administrators and security experts.

The Linux logging architecture employs a structured hierarchy of facilities and priorities, enabling precise control over the type and destination of logged data. Most modern Linux distributions integrate systemd's journald with traditional syslog services, ensuring compatibility while offering advanced features like structured data storage and searchability.

Administrators can fine-tune the Linux logging system through configuration files, allowing customization of log rotation, remote log management, and filtering options. This adaptability ensures that Linux logs remain invaluable for system monitoring, issue resolution, and auditing purposes. Linux systems dedicate a specific directory, /var/log, for storing logs. This directory houses a wide range of logs, including those generated by the operating system, services, and installed applications. It provides a centralized location for administrators to access critical information. Services such as PostgreSQL or Apache create their own log files in subdirectories within /var/log. For instance, if you're running an Apache server on a Debian-based Linux distribution, you can find its logs in the /var/log/apache2 directory. This directory typically contains separate files like access.log for tracking user requests and error.log for reporting issues.

Linux Logging Systems

Linux provides a comprehensive and flexible logging ecosystem with multiple systems that work together to capture, store, and manage system events and application logs. Understanding these different logging mechanisms is essential for effective system monitoring, troubleshooting, and security analysis.

Syslog Protocol

Syslog is a standardized protocol that forms the foundation of Linux logging. It provides a consistent format for system and application messages.

  • Standard RFC 5424
  • Port UDP 514, TCP 601
  • Format Timestamp, hostname, application name, PID, message
  • Features Message categorization by facility and priority

Rsyslog

Rsyslog is a modern, high-performance implementation of the syslog protocol that extends the basic functionality with advanced features.

  • Config /etc/rsyslog.conf
  • Modules Input, parsing, database, output modules
  • Templates Customizable message formats
  • Features Filtering, TCP/TLS, database output

Systemd Journal

Journald is part of systemd and provides structured binary logging with powerful querying capabilities for modern Linux distributions.

  • Config /etc/systemd/journald.conf
  • Storage Binary format in /var/log/journal/
  • Access Via journalctl command
  • Features Indexed fields, structured metadata

Logrotate

Logrotate manages log file rotation, compression, and removal to prevent logs from consuming excessive disk space.

  • Config /etc/logrotate.conf, /etc/logrotate.d/
  • Rotation Daily, weekly, monthly, or size-based
  • Actions Compression, truncation, mail delivery
  • Features Post-rotation scripts, retention policies

Rsyslog Configuration

The rsyslog daemon is configured through /etc/rsyslog.conf and additional files in /etc/rsyslog.d/. The configuration consists of modules, global directives, templates, and rules.

Example rsyslog.conf 
# Load modules
module(load="imuxsock") # provides support for local system logging
module(load="imklog")   # provides kernel logging support
module(load="imudp")    # UDP listener support
input(type="imudp" port="514")

# Global directives
$WorkDirectory /var/spool/rsyslog
$FileOwner root
$FileGroup adm
$FileCreateMode 0640
$DirCreateMode 0755
$IncludeConfig /etc/rsyslog.d/*.conf

# Rules (facility.priority action)
kern.*                          /var/log/kern.log
mail.*                          /var/log/mail.log
auth,authpriv.*                 /var/log/auth.log
*.emerg                         :omusrmsg:*

# Remote logging with TCP
*.warning                       @@logserver.example.com:514

Systemd Journal

The systemd journal provides structured logging for systems using systemd. Unlike traditional syslog, journald stores logs in a binary format with indexed fields for efficient searching and filtering.

Journald Configuration 
# /etc/systemd/journald.conf

[Journal]
Storage=persistent           # persistent, volatile, auto, none
Compress=yes                 # Compress with XZ
SystemMaxUse=4G              # Max disk space for journals
SystemKeepFree=1G            # Min free space to keep on disk
MaxFileSec=1month            # Max time to store entries
ForwardToSyslog=yes          # Send to syslog in parallel
Common journalctl Commands
# View all logs
$ journalctl
# View logs for a specific service
$ journalctl -u httpd.service
# Show logs from the last hour
$ journalctl --since "1 hour ago"
# Show only error and higher priority
$ journalctl -p err
# Follow new messages (like tail -f)
$ journalctl -f

Essential Log Analysis Commands

View Logs with Basic Tools
# Display entire log file
$ cat /var/log/syslog
# View log with paging
$ less /var/log/auth.log
# Show last 10 lines
$ tail /var/log/kern.log
# Follow new entries
$ tail -f /var/log/syslog
Filter and Search Logs
# Search for pattern in logs
$ grep "error" /var/log/syslog
# Case-insensitive search
$ grep -i "failed password" /var/log/auth.log
# Show context around matches
$ grep -A 2 -B 1 "critical" /var/log/syslog
# Find logs from a specific date
$ grep "Jan 15" /var/log/syslog
Advanced Log Processing
# Extract specific columns
$ awk '{print $1, $3, $5}' /var/log/syslog
# Count occurrences by field
$ awk '{print $5}' /var/log/syslog | sort | uniq -c | sort -nr
# Complex filtering
$ awk '/error/ && !/ignored/' /var/log/syslog
Journalctl Examples
# Filter by time range
$ journalctl --since "2023-01-15 12:00:00" --until "2023-01-15 13:00:00"
# Filter by unit and severity
$ journalctl -u nginx.service -p err
# Output formats
$ journalctl -o json-pretty
# Filter by field
$ journalctl _SYSTEMD_UNIT=ssh.service _UID=0

Linux Logging Best Practices

Security Recommendations

  • Enable logging for all authentication attempts (success and failure)
  • Configure centralized logging to a secure log server
  • Enable log integrity monitoring to detect tampering
  • Protect log files with appropriate permissions (0640 or stricter)
  • Implement encryption for logs transmitted over networks

Performance Optimization

  • Configure appropriate log rotation intervals based on system activity
  • Use compression for archived logs to save disk space
  • Implement rate limiting to prevent log flooding during attacks
  • Consider using faster storage for active log files
  • Monitor log storage space and set up alerts for low disk space

Log Management

  • Use consistent timestamps (preferably UTC) across all servers
  • Establish a log retention policy compliant with regulations
  • Categorize logs by type and importance for easier analysis
  • Implement structured logging formats where possible
  • Consider using log aggregation tools like Elasticsearch, Graylog, or Splunk
Pro Tip: While multiple logging systems can coexist on a Linux system, it's best practice to forward all logs to a single destination. Configure rsyslog to receive journal logs via $ModLoad imjournal or set ForwardToSyslog=yes in journald.conf for a unified logging solution.
