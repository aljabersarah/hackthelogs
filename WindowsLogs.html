Windows Event Logs

Windows Event Log Overview

Windows Event Logs are a critical component of the Windows operating system's logging and auditing framework, designed to provide comprehensive insights into system, security, and application activities. These logs capture detailed event data, including timestamps, process IDs, user account details, and system-level operations, enabling administrators and security professionals to monitor and analyze system behavior in real-time.

The Windows Event Log service efficiently records application, security, and system events into centralized log files. This functionality supports proactive system health monitoring, rapid identification of security incidents, and effective troubleshooting of configuration or operational issues.

Advanced security audit policies in Windows offer granular audit logging configurations, ensuring that critical activities such as logon attempts, privilege escalation, object access, and policy changes are captured with precision. By default, many of these policies are disabled, potentially creating blind spots in event monitoring. Enabling these policies enhances the ability to detect and investigate sophisticated attacks, such as privilege misuse, lateral movement, and unauthorized access. These audit logs are foundational for generating high-fidelity detections and facilitating forensic investigations.

Key Event Log Locations

Windows stores its event logs in several key locations depending on the log type:

Log Type Path Description
Application %SystemRoot%\System32\Winevt\Logs\Application.evtx Events logged by applications and programs
Security %SystemRoot%\System32\Winevt\Logs\Security.evtx Security audit events, login attempts, resource access
System %SystemRoot%\System32\Winevt\Logs\System.evtx System events from Windows components
Setup %SystemRoot%\System32\Winevt\Logs\Setup.evtx Events related to application setup
Forwarded Events %SystemRoot%\System32\Winevt\Logs\ForwardedEvents.evtx Events collected from remote computers
Pro Tip: Use Event Viewer ( eventvwr.msc) to browse these logs interactively, or PowerShell's Get-WinEvent cmdlet for scripted analysis.

Windows Advanced Audit Policies

Windows audit policy defines what types of events are written to the Security logs of your Windows servers. Establishing an effective audit policy helps you spot potential security problems, ensure user accountability and provide evidence in the event of a security breach. The recommended audit policy settings provided here are intended as a baseline for system administrators starting to define AD audit policies. You should be sure to consider the cybersecurity risks and compliance requirements of your organization. In addition, test and refine your policies before implementing them in your production environment.

PowerShell Commands for Log Analysis

Common PowerShell commands for querying and managing Windows Event Logs. These examples demonstrate how to retrieve, filter, and export event logs for system monitoring and troubleshooting.

Basic Event Log Query
PS> Get-WinEvent -LogName 'System' -MaxEvents 10
Retrieves the 10 most recent events from the System event log.
Filter by Error Level
PS> Get-WinEvent -FilterHashtable @{LogName='Application'; Level=2}
Retrieves all Error events (Level 2) from the Application log.
Levels: 1=Critical, 2=Error, 3=Warning, 4=Information
Filter by Date Range
PS> $startDate = (Get-Date).AddDays(-1) $endDate = Get-Date Get-WinEvent -FilterHashtable @{ LogName='System' StartTime=$startDate EndTime=$endDate Level=1,2 }
Retrieves all Critical and Error events from the System log in the last 24 hours.
Export to CSV
PS> Get-WinEvent -FilterHashtable @{LogName='System'; Level=1,2,3} -MaxEvents 100 | Select-Object TimeCreated, LevelDisplayName, ProviderName, Id, Message | Export-Csv -Path "C:\Logs\SystemErrors.csv" -NoTypeInformation
Exports the 100 most recent Critical, Error, and Warning events from the System log to a CSV file.
List Audit Policies
PS> auditpol /get /category:*
Displays the current audit policy settings for all categories configured on the system.
Clear Event Log
PS> Clear-EventLog -LogName "System" -Confirm:$false
Clears all events from the System event log without prompting for confirmation. Use with caution!
Check Event Log Size
PS> Get-WinEvent -ListLog * | Where-Object {$_.RecordCount -gt 0} | Select-Object LogName, RecordCount, FileSize, IsEnabled | Sort-Object RecordCount -Descending | Format-Table -AutoSize
Lists all event logs with their record count and size, sorted by number of records.
Find Failed Logon Attempts
PS> Get-WinEvent -FilterHashtable @{LogName='Security'; ID=4625} | Select-Object TimeCreated, @{Name='Username';Expression={$_.Properties[5].Value}}, @{Name='Source';Expression={$_.Properties[19].Value}}, @{Name='Status';Expression={$_.Properties[7].Value}}, @{Name='SubStatus';Expression={$_.Properties[9].Value}} | Format-Table -AutoSize
Retrieves failed logon attempts with username, source IP, and error codes.
Monitor Real-time Events
PS> $query = New-Object System.Diagnostics.Eventing.Reader.EventLogQuery("Security", [System.Diagnostics.Eventing.Reader.PathType]::LogName) $watcher = New-Object System.Diagnostics.Eventing.Reader.EventLogWatc her($query) Register-ObjectEvent -InputObject $watcher -EventName EventRecordWritten -Action { $event = $Event.SourceEventArgs.EventRecord Write-Host "New event: $($event.Id) at $($event.TimeCreated)" } $watcher.Enabled = $true
Sets up a real-time monitor for new events in the Security log.
Configure Audit Policy
PS> auditpol /set /subcategory:"Logon" /success:enable /failure:enable
Enables auditing of both successful and failed logon attempts.
Filter by Event ID
PS> Get-WinEvent -FilterHashtable @{ LogName='Security' ID=4624,4625,4634 }
Retrieves login events (successful logins, failed logins, and logouts) from the Security log.
Filter by Provider
PS> Get-WinEvent -FilterHashtable @{ LogName='System' ProviderName='Microsoft-Windows-Disk' Level=2,3 }
Retrieves all Error and Warning events related to disk operations.

Best Practices for Windows Event Logging

Follow these guidelines to ensure effective event logging in Windows environments:

01

Increase Log Size

Set larger default log sizes: Security (4GB), System (2GB), Application (2GB) to prevent log rotation during an incident.

02

Configure Retention

Set "Retention method" to "As needed" or "Overwrite events by days" rather than "Overwrite events as needed".

03

Centralize Logs

Forward critical events to a centralized SIEM solution for consolidated monitoring and analysis.

04

Advanced Audit Policies

Implement granular Advanced Audit Policy Configuration through Group Policy settings.

05

Regular Backups

Schedule regular backups of critical event logs to preserve evidence for forensic analysis.

06

Command Line Logging

Enable process command line auditing (Event ID 4688) to capture command-line arguments.

07

Monitor Log Clearing

Set alerts for log clearing activities (Event ID 1102) which may indicate tampering.

08

Critical Event Alerts

Configure real-time alerting for high-priority security events that require immediate attention.

Adversaries may use SID-History injection to gain additional privileges. The Windows security identifier (SID) is a unique value that identifies a user or group account. An account can hold additional SIDs in the SID-History attribute, allowing users to maintain access to network resources after being moved to a different domain.
